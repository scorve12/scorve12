### 2️⃣ **전략적 기술 선택 및 의사결정**

#### Django 프레임워크 선택 근거
| 고려사항 | Django 선택 이유 |
|----------|------------------|
| **비즈니스 확장성** | 향후 AI 기반 매칭/수요 예측 추가 예정 → Python 생태계 필수 |
| **개발자 채용** | Python 개발자 풀이 넓어 팀 확장 용이 |
| **개발 속도** | Django Admin으로 관리자 도구 빠르게 구축 |
| **마이그레이션** | 강력한 스키마 변경 관리 → 빠른 비즈니스 요구사항 대응 |
| **보안** | 검증된 보안 기능 (CSRF, XSS, SQL Injection 방어) |

#### PostgreSQL + PostGIS 선택
🎯 선택 이유:
✅ 농지 좌표, 경계선 등 공간정보 처리 필수
✅ 향후 드론 비행 경로 최적화 기능 확장 고려
✅ 오픈소스로 비용 절감


#### JWT 인증 전환 결정 (Critical Decision)
초기 상황:
✅ 세션 기반 인증 이미 구현 완료
❌ Flutter 팀: "모바일에서 세션 관리 복잡하다" 피드백

의사결정 프로세스:
1. 기술 검토: JWT vs Session 장단점 비교
2. 팀 논의: 프론트엔드 개발 난이도 vs 백엔드 변경 비용
3. 일정 분석: 1주일 소요 예상 vs 전체 일정 지연 위험
4. 결정: JWT로 전환 (프론트엔드 생산성 2배 향상)

📊 결과:
- 1주일 내 JWT 인증 전환 완료
- 프론트엔드 개발 속도 대폭 향상
- 모바일/웹 동일한 인증 체계로 일관성 확보

💡 교훈: "이미 만든 코드를 버리는 용기도 필요하다"

### 5️⃣ **실제 서비스 완성도 높이기**

#### 제3자 API 통합 (4개 서비스)
```
1️⃣ NICE API (본인인증)
- 암호화 방식: SEED + RSA 혼합 암호화
- 어려움: 복잡한 암호화 스펙 이해
- 해결: 테스트 환경에서 100+ 케이스 검증
- 시간: 1주일 소요

2️⃣ 토스페이먼츠
- 결제/취소/부분환불 플로우 구현
- 어려움: 결제 실패 시 재시도 로직, 이중 결제 방지
- 해결: Idempotency Key 패턴 적용
- 시간: 1.5주 소요

3️⃣ Naver Map API + VWorld API
- 어려움: 두 API의 좌표계 차이 (WGS84 vs GRS80)
- 해결: PostGIS ST_Transform으로 좌표 변환
- 시간: 3일 소요

4️⃣ 알리고 SMS API
- 발송 실패 시 재시도 로직 (3회까지)
- 비용 최적화: 중요 알림만 SMS, 나머지는 푸시
- 시간: 2일 소요

📊 API 통합 교훈:
- 문서만 믿지 말고 직접 테스트
- 엣지 케이스는 고객센터 문의 필수
- 커뮤니티/블로그 적극 활용
```

#### 크로스플랫폼 대응
```
Flutter (모바일) + React (웹) 동시 지원

🔧 기술적 챌린지:
1. 세션 → JWT 전환 (모바일 최적화)
2. 웹뷰 통합을 위한 전용 API 설계
3. 플랫폼별 다른 푸시 알림 (FCM vs APNs)

💡 해결책:
- API 버저닝 (v1, v2)
- 플랫폼 구분 헤더 (X-Platform: ios|android|web)
- 통합 푸시 서버 구축
```

#### 테스트 및 품질 관리
```
🧪 테스트 전략:
├── 단위 테스트: Django TestCase (커버리지 70%)
├── API 테스트: Postman Collection (40+ 케이스)
├── 베타 테스트: 실제 농민/조종사 50+ 참여
└── 부하 테스트: Locust (동시 접속 100명)

📊 베타 테스트 결과:
```
